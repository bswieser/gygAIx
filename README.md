# gygAIx
Building a virtual dungeon master. The project name is an homage to Gary Gygax, widely known as the progenator of the table top RPG "Dungeons and Dragons", substituting the 'a' for 'AI' for obvious comedic value.

## ⚠️ Repository Status: Conceptual Framework
**Please note: This repository contains Experiments, Examples, and Notes. It will NOT result in a fully buildable, plug-and-play repository.** This project is a collection of architectural blueprints, pipeline strategies, and code snippets designed to explore the creation of a fully autonomous AI Dungeon Master. 

## 📖 Project Overview
Project GygAIx (DM-Graph) aims to build a local, autonomous AI Dungeon Master that accurately simulates a tabletop RPG experience by reading, understanding, and enforcing rules exactly as written in published manuals. It moves beyond generic AI chatbots by refusing to hallucinate game mechanics or blend lore from unsupported systems. 

The system relies on a dual-engine architecture:
1.  **Document Ingestion & Extraction Engine:** Converts raw PDFs of rulebooks into a queryable Semantic Knowledge Graph.
2.  **AI DM Runtime:** A multi-agent framework orchestrated by a Finite State Machine (FSM) that uses the ingested graph to run the game, resolve mechanics, and generate narrative.

## 🗂️ Repository Structure
Because this is an active research and development space, the repository is organized into the following areas:

*   **`/Experiments`**: Isolated tests for computer vision and Layout parsing (e.g., using Qwen-VL and Docling to extract bounding boxes, handle multi-column layouts, and perform text/image classification).
*   **`/Examples`**: Sample JSON schemas for the Knowledge Graph representation, "Augmented Markdown" outputs, and mock Standard Operating Procedures (SOPs) that the AI generates to learn game rules.
*   **`/Notes`**: System design documents, architecture reviews, OCR pipeline strategies, and prompt engineering strategies for LLM routing (Gemma 3, Llama 3). 

## 🏗️ Core Architecture Concepts

### 1. Multi-Stage Document Pipeline
We utilize a multi-prompt, multi-model pipeline to decouple visual layout parsing from cognitive structuring:
*   **Phase 1 (Visual Triage):** Qwen-VL acts as the "eyes," identifying bounding boxes for text, tables, and images.
*   **Phase 2 (Semantic Graphing):** Gemma/Llama 3 rationalizes the layout into a structural reading order.
*   **Phase 3 (High-Fidelity Extraction):** Text is cleaned via OCR correction prompts, and images are passed back to local Vision-Language models to generate descriptive captions.

### 2. The Semantic Knowledge Graph (The "DM Brain")
RPG manuals are parsed into nodes (Tools, Processes, Entities, Themes) and edges (IS_A, HAS_A, CALCULATES). The content is dynamically classified into *Lore*, *Mechanics*, or actionable *Processes*. This graph operates locally via NetworkX and allows the AI to perform GraphRAG (Retrieval-Augmented Generation) without exceeding context windows.

### 3. FSM & Dynamic Tool Registry
The actual gameplay loop is handled by a LangGraph Finite State Machine. The AI processes intents through a strict loop: `SCENE_PHASE` -> `INTENT_EXTRACTION` -> `MECHANICS_RESOLUTION` -> `STATE_UPDATE` -> `NARRATIVE_GENERATION`. To bypass token limits, the AI relies on a Semantic Tool Registry, using vector search to only load the Python tools (like `roll_dice` or `query_table`) it needs at that exact moment.
